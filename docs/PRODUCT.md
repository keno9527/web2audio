# web2audio 产品需求设计

Chrome 插件负责提交当前网页文章，web2audio 服务端生成文章音频，love-song iOS 播放器承接文章音频的连续播放。

本文档用于产品评审和后续实现拆解，聚焦产品目标、系统边界、核心流程、功能范围、验收标准与待确认项。技术数据表、接口契约、存储方案和任务实现细节沉淀在技术方案文档中。

## 1. 背景与目标

### 背景

用户每天在 Chrome 浏览器中阅读大量文章，希望把值得消化的长文转成可听音频，并在 iOS 播放器中连续收听。当前诉求不是重新做一个泛用 TTS 工具，而是把「网页文章 -> 有声内容 -> 播放队列 -> iOS 连续播放」串成稳定闭环。

### 产品目标

| 维度 | 第一版目标 |
| --- | --- |
| 使用场景 | 在 Chrome 中主动保存文章，之后通过 iOS 播放器连续收听 |
| 用户范围 | 个人自用 MVP，优先跑通端到端体验 |
| 内容来源 | Chrome 当前网页文章，由插件在用户主动触发时提交 |
| 播放入口 | love-song iOS 播放器，复用其现有播放链路 |
| 完成标准 | 从 Chrome 提交文章，到音频生成完成，再到 iOS 播放器连续播放可稳定复现 |

## 2. 竞品与关键模块调研

### 代表产品

| 产品 | 相关能力 | 对本方案的启发 |
| --- | --- | --- |
| [Readwise Reader](https://readwise.io/read) | 稍后读、浏览器扩展、多端同步、TTS、文章/Newsletter/PDF/EPUB 等内容管理 | 「统一阅读收集箱 + 听读」是成熟方向 |
| [Speechify](https://speechify.com/) | Chrome 扩展、移动端、网页/文档 TTS、高质量语音 | 音色质量、语速和长文稳定性会直接影响收听体验 |
| [Matter](https://hq.getmatter.com/) | 稍后读、文章收集、音频播放队列 | 「保存后进入可听队列」比单次 TTS 更符合用户习惯 |
| [Instapaper](https://www.instapaper.com/) | 一键保存、全设备同步、离线阅读 | 离线和跨端能力是长期体验基础，但第一版先不展开 |

### 关键模块归纳

| 模块 | 第一版能力 | 第一版边界 |
| --- | --- | --- |
| 文章收集 | Chrome 插件一键提交当前网页 | 不自动监听全部浏览历史 |
| 正文提取 | 提取标题、正文、来源站点、作者、发布时间、封面图等可获得信息 | 不绕过付费墙，不由服务端抓取登录态内容 |
| 文章处理 | 正文清洗、长度校验、语言识别、重复提交识别 | 第一版按原始网页 URL 精确去重 |
| TTS 生成 | 稳定自然的单人声，支持长文处理和失败重试 | 不做多角色演播或复杂有声书效果 |
| 播放器接入 | 音频生成完成后进入 love-song 的固定歌单「今日待读」 | 不新增独立播放协议 |
| iOS 播放 | 在 love-song iOS 播放器内连续播放文章音频 | 第一版不覆盖后台播放、锁屏控制、CarPlay |

## 3. 与 love-song 播放器的关系

love-song 是独立播放器产品，web2audio 是独立的文章转音频服务。第一版依赖 love-song 已有 iOS 播放链路承接文章音频播放，但不直接改造 love-song 的产品定位为阅读器。

### 集成原则

| 原则 | 说明 |
| --- | --- |
| 系统解耦 | web2audio 负责文章提交、正文处理、TTS 生成和文章任务状态；love-song 负责可播放内容承接、歌单和播放体验 |
| 播放复用 | iOS 端播放、暂停、上一条、下一条、seek、播放历史等能力继续由 love-song 承接 |
| 固定歌单 | love-song 中存在系统托管歌单「今日待读」，用于收纳已生成完成的文章音频 |
| 顺序复用 | web2audio 只负责把可播放文章追加到「今日待读」末尾，不维护独立播放排序 |
| 状态分离 | 生成中和失败的文章由 web2audio 展示状态；只有可播放文章进入 love-song 播放链路 |

### 系统交互结果

文章音频生成完成后，web2audio 将音频登记为 love-song 可播放内容，并把它追加到「今日待读」歌单。iOS 用户在 love-song 中打开「今日待读」后，按照播放器现有顺序播放、随机播放、列表循环等策略收听文章音频。

## 4. 产品方案

### 总体方案

| Chrome 插件 | web2audio 服务端 | love-song iOS 播放器 |
| --- | --- | --- |
| 用户主动提交当前文章 | 接收文章、清洗正文、生成音频、维护任务状态 | 展示「今日待读」，连续播放已生成文章音频 |
| 展示提交结果和生成状态 | 重复提交时返回已有文章任务 | 复用既有播放器、歌单、播放历史和播放控制 |

### 核心流程

```mermaid
flowchart LR
  A["Chrome 当前文章"] -->|用户主动提交| B["文章任务"]
  B -->|正文清洗与校验| C["TTS 生成"]
  C -->|生成完成| D["登记为可播放内容"]
  D -->|追加到固定歌单| E["今日待读"]
  E -->|打开歌单| F["love-song iOS 播放器"]
  F -->|连续播放| G["文章音频收听"]
```

### 默认播放策略

- 「今日待读」是系统托管默认歌单。
- 文章音频生成完成后，自动追加到「今日待读」末尾。
- 先加入的文章先播放，后加入的文章排在后面。
- 播放顺序、随机播放、列表循环等行为复用 love-song 播放器现有策略。
- 生成中和失败的文章不会进入「今日待读」。

## 5. 功能模块

| 模块 | 第一版能力 | 非目标 |
| --- | --- | --- |
| Chrome 插件 | 一键提交当前文章，展示提交成功、已存在、生成中和失败状态 | 不自动抓取浏览历史，不批量导入书签 |
| 文章处理中心 | 按原始网页 URL 精确去重，清洗正文，识别语言，准备 TTS 输入 | 不合并不同 URL 但内容相同的文章 |
| TTS 任务 | 生成稳定自然的单人声文章音频，支持失败重试 | 不做多人对话、配乐、章节播报等复杂效果 |
| 播放器接入 | 可播放文章自动进入「今日待读」 | 不新增 web2audio 自有播放器 |
| iOS 文章体验 | 在 love-song iOS 播放器内播放文章音频，避免音乐化文案干扰 | 不要求第一版支持后台播放、锁屏控制或 CarPlay |

## 6. 内容与状态边界

### 文章内容语义

web2audio 在产品层管理文章音频任务，用于区分文章音频和普通音乐内容。文章任务保存文章来源、提取结果、生成状态、失败原因和对应的播放器承接结果。love-song 只承接已经生成完成、可以播放的文章音频。

| 内容状态 | 用户可见行为 |
| --- | --- |
| 已提交 | 插件或文章列表可看到任务已创建 |
| 处理中 | 用户可看到正文处理或音频生成正在进行 |
| 可播放 | 文章进入「今日待读」，可在 iOS 播放器中收听 |
| 失败 | 文章列表展示失败原因，并提供重试入口 |
| 已存在 | 重复提交同一原始网页 URL 时，返回已有文章任务和当前状态 |

### 文章字段语义

第一版产品层需要保留以下文章信息，用于列表展示、去重、生成和排障：

| 信息 | 用途 |
| --- | --- |
| 原始网页 URL | 重复提交识别和回看来源 |
| 标题 | 文章列表和播放页主标题 |
| 来源站点 | 列表和播放器副标题 |
| 作者 | 列表辅助信息，可为空 |
| 发布时间 | 文章来源信息，可为空 |
| 封面图 | 播放器展示，可为空 |
| 语言 | TTS 选择和清洗策略 |
| 正文处理状态 | 展示正文是否可用于生成 |
| 音频生成状态 | 展示是否生成完成、生成中或失败 |
| 失败原因 | 用户判断是否重试，排查生成问题 |
| 音频时长 | 播放器展示和收听预期 |
| 播放器承接结果 | 标识文章是否已进入 love-song 播放链路 |

### 插件提交边界

- 插件只在用户主动点击时提交当前网页。
- 插件提交原始网页 URL 和可提取正文信息。
- 服务端按原始网页 URL 精确去重。
- 重复提交时不创建新的文章任务，返回已有任务和当前状态。
- 不绕过付费墙，不由服务端模拟用户登录态抓取正文。
- 个人自用阶段可使用固定凭据或本地配置完成基础访问控制。

## 7. 验收标准与风险

### 验收标准

- [ ] Chrome 插件提交普通文章后，用户能看到明确的提交结果。
- [ ] 同一原始网页 URL 重复提交时，返回已有文章任务，不重复生成。
- [ ] 文章任务能从已提交或处理中流转到可播放或失败。
- [ ] 文章音频生成完成后，自动追加到 love-song 的「今日待读」歌单末尾。
- [ ] love-song iOS 播放器能从「今日待读」连续播放文章音频。
- [ ] 文章音频播放时不出现「未知艺术家」「首歌」等明显音乐化误导文案。
- [ ] 失败文章不进入「今日待读」，列表中可见失败原因并提供重试入口。
- [ ] 现有音乐播放功能不回退：曲库、歌单、播放控制和播放历史保持可用。

### 主要风险

| 风险 | 影响 | 建议处理 |
| --- | --- | --- |
| 网页正文提取失败 | 无法生成音频或内容质量差 | 插件端提取正文为主，服务端做清洗和兜底校验 |
| TTS 成本和时延 | 长文生成慢，影响可听内容及时进入歌单 | 让任务状态可见，失败后支持重试 |
| 播放器文案音乐化 | 文章音频在 iOS 中显示为歌曲，影响理解 | iOS 对文章音频展示文章标题、来源站点等内容语义 |
| 系统间接入失败 | 音频已生成但无法进入「今日待读」 | 保留接入失败状态和重试能力 |

## 8. 待确认项

| 问题 | 默认建议 | 影响 |
| --- | --- | --- |
| TTS provider 选择 | 先选稳定自然的云端单人声方案 | 影响音质、成本、生成速度和失败重试策略 |
| 文章正文保留策略 | 个人自用阶段保留正文和音频，支持手动删除 | 影响隐私、存储成本和问题排查能力 |
| 清理策略 | 第一版不自动清理，待使用频率稳定后再定义 | 影响存储空间和历史回听能力 |
| 访问控制 | 个人 MVP 先使用固定凭据或本地配置 | 影响插件提交和系统间接入安全边界 |
| iOS 文章频道入口 | 第一版优先使用「今日待读」歌单，文章频道视实现成本决定 | 影响 iOS 端导航和生成状态展示 |

> **第一版完成标准：** 从 Chrome 当前文章提交，到 web2audio 生成可播放文章音频，再到 love-song iOS 播放器通过「今日待读」连续播放的闭环可稳定复现。
